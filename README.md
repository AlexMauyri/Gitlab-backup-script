# gitlab-backup.sh

Bash-скрипт для автоматизированного бэкапа Git-репозиториев GitLab. Клонирует репозитории в режиме `--mirror`, при повторных запусках обновляет существующие зеркала, поддерживает параллельную обработку, повторные попытки при сетевых сбоях и уведомления в Telegram.

## Возможности

- Клонирование репозиториев в зеркальном режиме (`git clone --mirror`)
- Инкрементальное обновление уже существующих зеркал (`git remote update --prune`)
- Параллельная обработка репозиториев через несколько воркеров
- Повторные попытки при ошибках сети с экспоненциальной задержкой
- Список репозиториев можно задать прямо в конфиге или через отдельный файл
- Режим проверки доступности репозиториев без клонирования (`--test`)
- Логирование в файл и в stdout одновременно
- Уведомления об итогах бэкапа в Telegram
- Аутентификация через Personal Access Token (PAT)

## Требования

- Bash 4+
- `git`
- `curl` (только если используются уведомления в Telegram)
- Доступ на запись в директорию бэкапов и в директорию для логов

## Установка

```bash
chmod +x gitlab-backup.sh
```

Скрипт не требует установки — достаточно положить его в удобное место (например, `/opt/gitlab-backup/gitlab-backup.sh`) и запускать вручную или по cron.

## Использование

```bash
./gitlab-backup.sh [--config <файл>] [--test] [--help]
```

| Флаг               | Описание                                                   |
|--------------------|-------------------------------------------------------------|
| `--config <файл>`  | Путь к конфигурационному файлу                              |
| `--test`           | Проверить доступность репозиториев без клонирования         |
| `--help`           | Показать справку и шаблон конфигурационного файла           |

### Поиск конфигурационного файла

Если флаг `--config` не указан, скрипт ищет конфиг в следующем порядке:

1. Файл, переданный через `--config`
2. `~/.config/gitlab-backup.conf`
3. `/etc/gitlab-backup.conf`

Если ни один файл не найден — скрипт завершится с ошибкой.

## Конфигурационный файл

Конфиг — это обычный shell-файл (`source`), переменные задаются как `KEY=value`.

### Обязательные параметры

| Параметр       | Описание                                                       |
|----------------|------------------------------------------------------------------|
| `GITLAB_TOKEN` | Personal Access Token с правом `read_repository`                 |
| `BACKUP_DIR`   | Абсолютный путь к директории, куда складываются бэкапы            |

Также обязательно нужно указать **либо** `REPOS`, **либо** `REPO_FILE` (можно оба одновременно).

### Список репозиториев

| Параметр    | Описание                                                                 |
|-------------|---------------------------------------------------------------------------|
| `REPOS`     | Список URL репозиториев, разделённых символом `\|`                        |
| `REPO_FILE` | Путь к файлу со списком URL, один URL на строку. Поддерживаются комментарии через `#` |

### Опциональные параметры

| Параметр             | По умолчанию              | Описание                                                |
|----------------------|----------------------------|-----------------------------------------------------------|
| `PARALLEL_JOBS`      | `4`                        | Количество параллельных воркеров                            |
| `RETRY_COUNT`        | `3`                        | Количество повторных попыток при сетевых ошибках             |
| `LOG_FILE`           | `./logs/gitlab-backup.log` | Путь к файлу логов                                          |
| `TELEGRAM_BOT_TOKEN` | —                          | Токен Telegram-бота для уведомлений                          |
| `TELEGRAM_CHAT_ID`   | —                          | ID чата/канала, куда отправлять уведомления                  |

Уведомления в Telegram отправляются только если заданы и `TELEGRAM_BOT_TOKEN`, и `TELEGRAM_CHAT_ID`.

### Пример конфигурационного файла

```bash
# /etc/gitlab-backup.conf

GITLAB_TOKEN=glpat-xxxxxxxxxxxxxxxxxxxx
BACKUP_DIR=/backup/gitlab

REPOS=https://gitlab.com/group/project1.git|https://gitlab.com/group/project2.git
# или
# REPO_FILE=/etc/gitlab-repos.list

PARALLEL_JOBS=4
RETRY_COUNT=3
LOG_FILE=/var/log/gitlab-backup.log

TELEGRAM_BOT_TOKEN=123456:ABC-DEF1234
TELEGRAM_CHAT_ID=-123456789
```

Полный шаблон конфига также можно получить командой:

```bash
./gitlab-backup.sh --help
```

### Формат файла со списком репозиториев (`REPO_FILE`)

```text
# Комментарии и пустые строки игнорируются
https://gitlab.com/group/project1.git
https://gitlab.com/group/project2.git  # тоже можно с комментарием в конце строки
```

## Как это работает

1. Скрипт находит и загружает конфиг, валидирует обязательные параметры.
2. Собирает список репозиториев из `REPOS` и/или `REPO_FILE`, убирает дубликаты.
3. Добавляет в каждый URL аутентификацию через `oauth2:<TOKEN>@`.
4. Делит список репозиториев на чанки по числу `PARALLEL_JOBS` и запускает воркеры параллельно.
5. Каждый воркер для каждого репозитория:
   - если локальное зеркало уже существует — выполняет `git remote update --prune`;
   - если нет — клонирует репозиторий через `git clone --mirror`.
   - При ошибке повторяет попытку до `RETRY_COUNT` раз с экспоненциальной задержкой (2, 4, 8... секунд).
6. После завершения всех воркеров собирает статистику (успехи/ошибки), пишет итог в лог и отправляет сводку в Telegram (если настроено).

### Структура бэкапов

Репозитории сохраняются по пути:

```
$BACKUP_DIR/<project_name>/<DD-MM-YYYY>/<project_name>
```

При повторном запуске в тот же день существующее зеркало обновляется, а не клонируется заново.

## Режим проверки (`--test`)

```bash
./gitlab-backup.sh --config ./gitlab-backup.conf --test
```

В этом режиме скрипт:

- загружает конфиг и собирает список репозиториев;
- выполняет `git ls-remote --heads` для каждого URL, чтобы проверить доступность;
- выводит итоговую статистику (сколько доступно / недоступно);
- **не клонирует и не обновляет** ни один репозиторий.

## Логирование

Весь вывод скрипта дублируется в файл, указанный в `LOG_FILE` (директория создаётся автоматически), и в stdout/stderr. Каждая строка лога содержит таймстемп и уровень (`INFO`, `WARN`, `ERROR`).

## Коды завершения

| Код   | Значение                                              |
|-------|----------------------------------------------------------|
| `0`   | Все репозитории успешно забэкаплены                      |
| `1`   | Один или несколько репозиториев не удалось обработать, либо ошибка конфигурации |
| `130` | Прервано сигналом `SIGINT` (Ctrl+C)                       |
| `143` | Прервано сигналом `SIGTERM`                               |

При ненулевом коде завершения (если настроен Telegram) отправляется уведомление о прерывании.

## Запуск по cron

Пример ежедневного запуска в 03:00:

```cron
0 3 * * * /opt/gitlab-backup/gitlab-backup.sh --config /etc/gitlab-backup.conf >> /var/log/gitlab-backup-cron.log 2>&1
```

## Безопасность

- Personal Access Token хранится в конфигурационном файле в открытом виде — ограничьте права на файл (`chmod 600`).
- Используйте токен с минимально необходимым правом `read_repository`.
- Не добавляйте конфигурационный файл с токеном в систему контроля версий.

## Известные ограничения

- Скрипт рассчитан на Linux/Unix-окружение с Bash 4+ (используются ассоциативные массивы и `mapfile`).
- Параллелизм реализован на уровне процессов (фоновые воркеры), а не через `xargs`/`parallel`.
